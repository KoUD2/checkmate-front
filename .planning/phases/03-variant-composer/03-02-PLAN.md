## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| controller→service | Service receives validated DTO data from the controller layer. Controller (Plan 03) applies `ValidationPipe(whitelist:true)`. Service still defends against logically-invalid input (e.g., duplicate taskIds) because validation cannot catch app-level rules. |
| service→database | All DB writes use Prisma parameterized queries. No string interpolation in queries. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | AssignTasksDto with malicious taskIds (SQL keywords, path traversal strings) | mitigate | `@IsUUID('all', { each: true })` rejects anything that is not a valid UUID at DTO layer; Prisma parameterizes the eventual SQL — no string concatenation. |
| T-03-05 | Denial of Service | `assignTasks` with very large taskIds array (e.g., 100,000 entries) exhausts transaction time | accept | Phase 3 use case is admin-only with realistic exam-bank sizes (~50-100 tasks max). No explicit cap added; defer to RESEARCH §Pitfall 4 deduplicate check which is O(n). |
| T-03-06 | Tampering | Duplicate `examTaskId` in taskIds array triggers `@@unique([variantId, examTaskId])` DB error mid-transaction | mitigate | Service deduplicates BEFORE `$transaction`: `if (new Set(taskIds).size !== taskIds.length) throw new BadRequestException(...)`. Verified by service.spec test "throws BadRequestException when taskIds contains duplicates". |
| T-03-07 | Information Disclosure | Unpublished variants returned by `listPublished()` to students | mitigate | `listPublished()` applies `where: { published: true }` to BOTH `findMany` and `count`. Verified by service.spec test "listPublished filters where published true (VARIANT-04)". Admin endpoint is a separate method `adminList()` (no published filter) — Plan 03 places it on the admin-guarded controller. |
| T-03-08 | Elevation of Privilege | Service has no auth check; relies on controller-level RolesGuard | mitigate | Plan 03 wires `JwtAuthGuard + RolesGuard + @Roles(Role.ADMIN)` at class level on `VariantsAdminController`. Service itself trusts caller because Phase-2 D-03 convention is class-level guards only. |
| T-03-09 | Tampering | Mass-assignment via UpdateVariantDto setting fields not intended (e.g., `id`, `createdAt`) | mitigate | DTO only declares `title`, `description`, `published`. Global `ValidationPipe(whitelist: true)` in `main.ts` strips unknown fields. PartialType preserves whitelist. |
</threat_model>
